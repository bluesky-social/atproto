# Email management work on `msi/account-manager-reset-email` — reference for adding handle change

This document summarises the changes shipped on the current branch (vs `main`) so the
same skeleton can be reused for **changing a user's handle from the OAuth account
manager UI**. The UX will differ a bit (handles are public and don't need a token
mailed to a previous handle), but the architectural seams are the same.

The branch adds, end‑to‑end:

1. New OAuth API endpoints in `@atproto/oauth-provider-api`
2. New `AccountStore` methods + `AccountManager` flows + lifecycle hooks in
   `@atproto/oauth-provider`
3. PDS implementation of those store methods on top of the existing
   `accountManager.*` helpers (`packages/pds/src/account-manager/oauth-store.ts`)
4. A reshape of the underlying PDS `accountManager` helpers and the matching
   `com.atproto.server.*` xrpc endpoints
5. New UI views, data hooks, routes, and an account‑page banner
6. Locale entries (50+ `.po` files, all auto‑generated by lingui — don't write
   these by hand)
7. End‑to‑end browser tests

The list of files touched is in [.changeset/sweet-bobcats-rescue.md](.changeset/sweet-bobcats-rescue.md)
and `git diff main...HEAD --stat`.

---

## 1. `@atproto/oauth-provider-api` — wire format

[packages/oauth/oauth-provider-api/src/api-endpoints.ts](packages/oauth/oauth-provider-api/src/api-endpoints.ts)
is the single source of truth for the request/response shapes between the React
UI and the provider middleware.

For email, four endpoints were added to the `ApiEndpoints` map:

```ts
'/update-email-request':  { method: 'POST', input: InitiateEmailUpdateInput,       output: InitiateEmailUpdateOutput }
'/update-email-confirm':  { method: 'POST', input: ConfirmEmailUpdateInput,        output: { success: true } }
'/verify-email-request':  { method: 'POST', input: InitiateEmailVerificationInput, output: { success: true } }
'/verify-email-confirm':  { method: 'POST', input: ConfirmEmailVerificationInput,  output: { success: true } }
```

…with matching `Input/Output` type aliases defined further down the file.

**For the handle update**: the file already exposes `'/verify-handle-availability'`
(used by sign‑up). You'll likely want to add at minimum:

- `'/update-handle'` (or `'/update-handle-request'` + `'/update-handle-confirm'`
  if you decide to require a confirmation step) with an `Input` shaped
  `{ sub, handle, locale? }`. Note `sub` is always present on the authenticated
  endpoints — see the auth middleware below.

Conventions to follow:

- Endpoint paths are `kebab-case`, prefixed with `/`.
- `sub: string` is the standard discriminator on authenticated endpoints — DO
  NOT use `did`.
- Optional `locale?: string` is included on any input that may produce a mail
  template. (Handle change probably doesn't need it unless you mail a confirmation.)

## 2. `@atproto/oauth-provider` — store, manager, hooks, middleware

### 2.1 `AccountStore` interface — [packages/oauth/oauth-provider/src/account/account-store.ts](packages/oauth/oauth-provider/src/account/account-store.ts)

Each new endpoint added a method to the `AccountStore` interface AND to the
`isAccountStore` runtime checker array. The pattern for email was:

```ts
updateEmailRequest(data: UpdateEmailRequestInput): Awaitable<UpdateEmailRequestOutput>
updateEmailConfirm(data: UpdateEmailConfirmInput): Awaitable<Account | null>
verifyEmailRequest(data: VerifyEmailRequestInput): Awaitable<void>
verifyEmailConfirm(data: VerifyEmailConfirmInput): Awaitable<Account | null>
```

Key points:
- `*Confirm` methods return `Account | null`. Returning `null` is the *only* way
  the manager throws `InvalidRequestError('Invalid token')` — see
  `account-manager.ts`. So in PDS, intercept the matching xrpc errors and
  translate them to `null`.
- Type aliases re‑export the `*Input/*Output` shapes from
  `@atproto/oauth-provider-api`, e.g.
  `export type UpdateEmailRequestInput = InitiateEmailUpdateInput`. Keep this
  pattern.
- Don't forget to add the new method names to the `isAccountStore` array, or
  runtime store validation will silently miss the regression.

**For the handle update**, plan one or two methods, mirroring the shape:
e.g. `updateHandleRequest(...)` / `updateHandleConfirm(...)`, or just a single
`updateHandle(...)` if there's no two‑step confirmation.

### 2.2 `AccountManager` — [packages/oauth/oauth-provider/src/account/account-manager.ts](packages/oauth/oauth-provider/src/account/account-manager.ts)

Each new public method follows the same template:

```ts
public async updateEmailRequest(deviceId, deviceMetadata, input, account) {
  await this.hooks.onChangeEmailRequest?.call(null, { deviceId, deviceMetadata, input, account })
  const result = await this.store.updateEmailRequest(input)
  await this.hooks.onChangeEmailRequested?.call(null, { ... })
  return result
}
```

Always: pre‑hook → store call → post‑hook (`onXxx` before, `onXxxed` after).
The `account: Account` is passed in because these endpoints are *authenticated*
(see middleware below), so `account` is guaranteed.

The branch also did a small refactor of the surrounding code (unrelated to
email per se) that you'll inherit: `signUp` / `authenticateAccount` /
`resetPasswordRequest` / `resetPasswordConfirm` were rewritten to use a new
`callAsync` util and to put their `constantTime` wrapping consistently around
the hook calls. There's no need to undo or repeat that work — it just exists
on this branch.

A new util was added at
[packages/oauth/oauth-provider/src/lib/util/function.ts](packages/oauth/oauth-provider/src/lib/util/function.ts):
`callAsync(fn, ...args)` — handy when you want to call a possibly‑undefined
member function without binding `this`. Reuse it.

### 2.3 OAuth lifecycle hooks — [packages/oauth/oauth-provider/src/oauth-hooks.ts](packages/oauth/oauth-provider/src/oauth-hooks.ts)

For each new feature, add a `before` and `after` hook to the `OAuthHooks` type.
Email added 8 hooks (4 endpoints × 2). All are typed as
`(data: { input, deviceId, deviceMetadata, account: Account }) => Awaitable<void>`.

For handle change, add equivalent: `onChangeHandleRequest` / `onChangeHandleRequested`
and confirm variants if you go two‑step, or just `onUpdateHandle`/`onUpdatedHandle`.

Naming convention spotted in existing code (slightly inconsistent — be aware):
- `onChangeEmailRequest` / `onChangeEmailRequested` (request side)
- `onUpdateEmailConfirm` / `onUpdateEmailConfirmed` (confirm side)
- `onVerifyEmailRequest` / `onVerifyEmailRequested`
- `onVerifyEmailConfirm` / `onVerifyEmailConfirmed`

Pick one prefix (`onChangeHandle…` or `onUpdateHandle…`) and stay consistent
within your endpoint set.

### 2.4 API middleware — [packages/oauth/oauth-provider/src/router/create-api-middleware.ts](packages/oauth/oauth-provider/src/router/create-api-middleware.ts)

Each endpoint added a `router.use(apiRoute({...}))` block. They all call
`authenticate.call(this, req, res)` to get `{ account }`, validate `this.input`
against a Zod schema (use the `subSchema`, `emailSchema`, `emailOtpSchema`,
`localeSchema` already exported there — add a `handleSchema` if missing), then
delegate to `server.accountManager.<method>(this.deviceId, this.deviceMetadata,
this.input, account)` and return `{ json: { ... } }`.

Note: the schema uses `.strict()` to reject unknown fields. Keep that.

## 3. PDS implementation

### 3.1 `AccountManager` (PDS) — [packages/pds/src/account-manager/account-manager.ts](packages/pds/src/account-manager/account-manager.ts)

**Major shape change**: the email helpers were moved up from the xrpc handlers
into the PDS `AccountManager` class so they can be called from BOTH the legacy
xrpc handlers (`com.atproto.server.*`) AND the new oauth-provider store
(`oauth-store.ts`). The same will apply to handle: today
[packages/pds/src/api/com/atproto/identity/updateHandle.ts](packages/pds/src/api/com/atproto/identity/updateHandle.ts)
contains business logic (PLC update, account lookup, sequencer event). Move
that logic into a method on `AccountManager` so the OAuth store can reuse it.

Notable details:
- The `AccountManager` constructor now takes `mailer: ServerMailer` as a new
  arg ([packages/pds/src/context.ts](packages/pds/src/context.ts)), because
  the consolidated helpers send mail directly. Follow the same pattern if the
  handle flow needs to mail anything.
- `confirmEmail` was changed from `{ did, token }` to `(did, email, token)` and
  it now does the full email‑match check inside (previously the xrpc handler
  did it).
- `updateEmail` was renamed to `updateAccountEmail` (the low‑level db update)
  and a new `updateEmail(did, email, token, opts)` was introduced as the
  high‑level entry point that does validation + token check + optionally sends
  the post‑change confirmation mail. Watch for this rename — the admin endpoint
  [packages/pds/src/api/com/atproto/admin/updateAccountEmail.ts](packages/pds/src/api/com/atproto/admin/updateAccountEmail.ts)
  was updated to call the new low‑level name.
- `UserAlreadyExistsError`
  ([packages/pds/src/account-manager/helpers/account.ts](packages/pds/src/account-manager/helpers/account.ts))
  was given a default message and a `name` so that the OAuth store can
  `instanceof`-check it and translate to a UI error. There's an analogous
  `HandleUnavailableError` you can leverage.

### 3.2 OAuth store (PDS) — [packages/pds/src/account-manager/oauth-store.ts](packages/pds/src/account-manager/oauth-store.ts)

This is where the OAuth provider's `AccountStore` interface gets implemented
in terms of the PDS `accountManager`. New methods (`verifyEmailRequest`,
`verifyEmailConfirm`, `updateEmailRequest`, `updateEmailConfirm`) follow this
template:

```ts
async updateEmailConfirm({ sub: did, token, email, locale }: UpdateEmailConfirmInput) {
  assert(isDidString(did), 'sub must be a valid DID string')
  try {
    const account = await this.accountManager.updateEmail(did, email, token, {
      sendConfirmationEmail: true,
      locale,
    })
    return this.buildAccount(account)
  } catch (cause) {
    if (cause instanceof UserAlreadyExistsError) {
      throw new InvalidRequestError(cause.message, cause)
    }
    throw cause
  }
}
```

Conventions:
- `assert(isDidString(sub), ...)` — there's a TODO upstream to type `Sub` as
  `DidString`. Repeat the assert; don't try to fix it on this branch.
- Wrap the call in a try/catch; translate domain errors (`UserAlreadyExistsError`,
  `HandleUnavailableError`, `XrpcAuthRequiredError`, `XrpcInvalidRequestError`)
  into either `InvalidRequestError` (if it should bubble up to the user) or
  `null` (if the manager should map it to "Invalid token").
- Always return through `this.buildAccount(...)` for `Account | null` returns.

### 3.3 xrpc endpoints (PDS) — `packages/pds/src/api/com/atproto/server/*`

Each xrpc endpoint that overlaps with an OAuth API endpoint was rewritten with
this structure:

1. Hoist a shared `auth` factory (e.g. `requestEmailUpdateAuth(ctx)`) so both
   the request and confirm endpoints use the same authorization rules.
2. Hoist `rateLimit` config to a constant.
3. Branch on `entrywayClient`: in the entryway path, just forward the call;
   in the local PDS path, call the new `accountManager` method.

For email‑update specifically, `requestEmailUpdateAuth` uses
`scopes: ACCESS_FULL` and *throws* in `authorize:` to reject all OAuth
credentials — only password sessions are allowed to change email. The
underlying intent is **"email change is a high‑security action; route it
through the account manager UI rather than third-party OAuth apps"**. Apply the
same logic to handle: think hard about whether handle change should be
allowed via OAuth scopes or only via the account manager UI.

The `confirmEmail.ts` and `requestEmailConfirmation.ts` files now import their
auth factory from each other to keep them in lock‑step. Mimic this if you split
handle change into request/confirm endpoints.

The PDS context now wires the `mailer` into `accountManager`
(see [packages/pds/src/context.ts](packages/pds/src/context.ts)).

### 3.4 Mailer — [packages/pds/src/mailer/index.ts](packages/pds/src/mailer/index.ts)

`sendConfirmEmail` and `sendUpdateEmail` now accept an optional `locale` field
in their `params` (currently a TODO — the templates ignore it). If the handle
flow doesn't email anything, you can skip this. If it does, mirror the same
locale‑forwarding TODO.

## 4. UI — `packages/oauth/oauth-provider-ui`

### 4.1 The `Api` class — [packages/oauth/oauth-provider-ui/src/lib/api.ts](packages/oauth/oauth-provider-ui/src/lib/api.ts)

`JsonClient` was extended with an `onFetchSuccess` map that lets the session
context update local cached state in response to specific endpoints. See
[packages/oauth/oauth-provider-ui/src/lib/json-client.ts](packages/oauth/oauth-provider-ui/src/lib/json-client.ts)
and the `'/update-email-confirm'` / `'/verify-email-confirm'` entries in
[packages/oauth/oauth-provider-ui/src/contexts/session.tsx](packages/oauth/oauth-provider-ui/src/contexts/session.tsx).
**You will almost certainly want a `'/update-handle'` entry** that calls
`updateAccount(input.sub, { preferred_username: input.handle })`. The
`updateAccount` helper was added in this branch — it patches one session in
the in‑memory list.

The `Api` class also gained method shortcuts (`signIn`, `signOut`, …). The
older `doSignIn` / `doSignOut` style on the session context are now marked
`@deprecated`. **For new code, expose new operations via methods on `Api` and
hooks in `data/`, not on the session context.**

### 4.2 React‑Query hooks — [packages/oauth/oauth-provider-ui/src/data/email.ts](packages/oauth/oauth-provider-ui/src/data/email.ts)

This file is the new canonical pattern (compare with the simpler
[packages/oauth/oauth-provider-ui/src/data/password.ts](packages/oauth/oauth-provider-ui/src/data/password.ts)).
Each operation is a `useMutation` that:

- forwards the `locale` from `useCurrentLocale()` if not provided
- shows a success notification via `useNotificationsContext().notify`
- shows an error notification + `console.error` on failure

For handle, add `packages/oauth/oauth-provider-ui/src/data/handle.ts` with
`useUpdateHandle()` (and request/confirm split if applicable) plus possibly
`useVerifyHandleAvailability()` (a hook wrapper around the existing
`api.validateHandleAvailability(...)` for the live‑validation UX).

The reusable email components are good models:

- [packages/oauth/oauth-provider-ui/src/components/update-email-view.tsx](packages/oauth/oauth-provider-ui/src/components/update-email-view.tsx) —

  [verify-email-confirm-form.tsx](packages/oauth/oauth-provider-ui/src/components/verify-email-confirm-form.tsx) —

  optional cancel handler

For handle, you'll likely want an `UpdateHandleView` with an
`InputHandleAvailability` field (live availability check) — there's already
`verify-handle-availability` wiring on the API side, and the existing handle
input from sign‑up
([components/sign-up-account-form.tsx](packages/oauth/oauth-provider-ui/src/components/sign-up-account-form.tsx))
can probably be lifted into a reusable component.

A small button factory was added:
[components/forms/button-request-reset.tsx](packages/oauth/oauth-provider-ui/src/components/forms/button-request-reset.tsx).
It wraps `ButtonCooldown` with the paper‑plane icon and "Send reset code"
default. The cooldown hook
([hooks/use-rate-limited-action.ts](packages/oauth/oauth-provider-ui/src/hooks/use-rate-limited-action.ts))
got a small refactor to take `{ action, cooldownSeconds, initialCooldown }` as
options instead of a positional `action`.

### 4.4 Pages and routing — [packages/oauth/oauth-provider-ui/src/pages/account/(authenticated)/route.tsx](packages/oauth/oauth-provider-ui/src/pages/account/(authenticated)/route.tsx)

Two new pages were registered in `DEFAULT_PAGES`:

```ts
'/email':        { icon: EnvelopeIcon,    position: 40, title, description, component }
'/email/verify': { hidden: true, icon: ShieldCheckIcon, position: 45, ... }
```

`hidden: true` keeps the page out of the side nav but still routable. **Add a
`'/handle'` entry** with appropriate icon (`AtIcon`, `IdentificationCardIcon`,
something from `@phosphor-icons/react`) and a position that fits the order.
Existing positions: `0=index`, `30=password`, `40=email`, `50=about`. Pick e.g.
`35` for handle.

The route also gained an `<Admonition>` banner that renders when
`account.email && !account.email_verified`. **For handle**, you probably do
*not* need this banner — handle isn't "verified" the same way — unless you
introduce some "handle is unconfirmed" state.

The page files
[pages/account/(authenticated)/email/page.tsx](packages/oauth/oauth-provider-ui/src/pages/account/(authenticated)/email/page.tsx)
and [email/verify/page.tsx](packages/oauth/oauth-provider-ui/src/pages/account/(authenticated)/email/verify/page.tsx)
are thin adapters that pull `{ account }` from `useAuthenticatedSession()` and
wire the React‑Query hooks into the view component. Use them as the template
for `pages/account/(authenticated)/handle/page.tsx`.

### 4.5 Account selector mock — [packages/oauth/oauth-provider-ui/account-page.html](packages/oauth/oauth-provider-ui/account-page.html)

This standalone HTML file mocks the API for local development outside of the
PDS. Every new endpoint needs a corresponding `case` in the `switch` block
inside `fetch`. Note especially how it mutates the in‑memory `accounts` map
to mirror the real backend semantics — when handle change lands, the mock
must update `preferred_username` on the account.

## 5. Locales

Files under [packages/oauth/oauth-provider-ui/src/locales/](packages/oauth/oauth-provider-ui/src/locales/)
were updated for every supported language. **Don't write these by hand** —
add the new strings inline (`<Trans>…</Trans>`, `t\`…\``, `msg\`…\``) and run
the lingui extract script to regenerate the `.po` files. The branch only
modified `messages.po` — check the package scripts for the right command.

## 6. Tests — [packages/pds/tests/account-manager.test.ts](packages/pds/tests/account-manager.test.ts)

Two new e2e tests were added:

- "allows validating the email address" — clicks the verification banner,
  mocks `sendConfirmEmail`, types the token, asserts success copy
- "allows changing the email address" — drives the full flow, mocks both
  `sendUpdateEmail` and `sendConfirmEmail`, checks both tokens are processed,
  asserts post‑change verification works

Use the same browser‑based `PageHelper` style for the handle test. Strings are
asserted in **French** because the test runs with `languages = ['fr']` — when
you add new UI text, the French translation must be wired in (or the lingui
default, depending on test setup).

The pattern `using sendXxxMock = jest.spyOn(...).mockImplementation(...)`
relies on TC39 explicit resource management and auto‑restores the spy at scope
end — keep using it.

## 7. Recommended ordering for the handle PR

1. Add the new endpoint(s) to `@atproto/oauth-provider-api`.
2. Extend `AccountStore` (interface + `isAccountStore` array) and add the
   manager methods + hook calls in `@atproto/oauth-provider`.
3. Add the API middleware route(s) with Zod validation.
4. Move the existing handle update logic from
   `packages/pds/src/api/com/atproto/identity/updateHandle.ts` into
   `packages/pds/src/account-manager/account-manager.ts` so both the xrpc
   handler and the OAuth store can call it. Refactor `updateHandle.ts` to
   delegate, with the same `entrywayClient` branching pattern used by the
   email endpoints.
5. Implement the new methods in `packages/pds/src/account-manager/oauth-store.ts`.
6. Add the data hooks in `packages/oauth/oauth-provider-ui/src/data/handle.ts`.
7. Add `Api` method shortcuts and an `onFetchSuccess` entry that updates
   `preferred_username` on the cached session.
8. Add the view components and the page.
9. Register the route in `route.tsx`.
10. Update the `account-page.html` mock.
11. Regenerate locales.
12. Add an e2e test mirroring the email one.
13. Add a changeset.
